## 1. システム全体像

### 1-1. ブロック構成（入力 → 処理 → 出力）

> ※ Arduinoを中心に「何が入力で何が出力か」をテキストの図で整理します。

**記入例：**
```
【入力】                 【Arduino UNO R3】            【出力】
タクトスイッチ ─────→┐                         ┌──→ LED（赤・緑）
超音波センサー ─────→┤   状態管理・判定処理    ├──→ パッシブブザー
                      └─────────────────────────┘
```

**↓ 自分のシステムを書いてください**
```
【入力】                   【Arduino UNO R3】              【出力】
ジョイスティック（X/Y） ─→┐                           ┌→ LEDマトリクス（ゲーム画面）
ジョイスティック押込み ─→┤   状態管理・ゲーム処理     ├→ ブザー（効果音）
                           └───────────────────────────┘
```

補足説明（図で表現しきれない場合）：
- ジョイスティックのアナログ値から移動方向を決定
- ゲーム状態（待機/プレイ/ゲームオーバー）に応じて表示と音を制御

### 1-2. 状態遷移（ガジェットの「ふるまい」設計）

> ※ ガジェットが「どんな状態を持つか」を定義します。
> この設計が loop() の骨格になります。

**記入例：**
```
[電源ON / 初期化]
  ↓（初期化完了）
[待機中] ──（センサー検知）──→ [計測中] ──（閾値超え）──→ [警告中]
   ↑                                                            │
   └────────────────────（リセットボタン）────────────────────────┘
```

**↓ 自分の状態遷移を書いてください**
```
[電源ON / 初期化]
  ↓（初期化完了）
[待機（タイトル）] ──（押し込み）──→ [プレイ中] ──（敵と衝突）──→ [ゲームオーバー]
   ↑                                                        │
   └──────────────（押し込み）───────────────┘

```

**各状態の意味：**

| 状態名 | この状態のとき、何をしているか | LEDやブザーなど出力の様子 |
|:--|:--|:--|
| 初期化 | ライブラリ初期化、変数初期化 | 全消灯 |
| 待機（タイトル） | ゲーム開始待ち、難易度設定（任意） | タイトル表示（固定/点滅） |
| プレイ中 | 入力に応じて移動、敵生成・移動、衝突判定 | 画面表示＋必要に応じて効果音 |
| ゲームオーバー | 終了表示、リセット待ち | 全体点滅＋ゲームオーバー音 |

> [!TIP]
> すべての状態から「抜け出せるルート」があるか確認してください。
> 「このボタンを押せば必ずリセットに戻る」が保証されていますか？

### 2-3. タイミング設計（delay vs millis）

> ※ `delay(N)` はN ms間、**すべての処理**が止まります。
> 複数の処理を並行させたい場合は `millis()` を使います。

| 処理 | 必要な周期 | delay / millis | 判断の理由 |
|:--|:--|:--|:--|
| プレイヤー移動 | 100ms | **millis** | 入力監視・描画と並行するため |
| 敵移動 | 300ms（難易度で変更） | **millis** | ゲーム速度を制御しつつ並行動作させるため |
| 描画更新 | 50ms | **millis** | ちらつき抑制・滑らかに見せるため |
| 敵点滅（区別） | 150ms | **millis** | 単色表示でも敵を見分けるため |

> [!CAUTION]
> `delay()` を使う場合、その間はボタン入力もセンサー読み取りも**すべて止まります**。

## 7. AIレビュー記録

> グループレビューの前に必ず実施してください。

### Q1: 設計の妥当性確認

> 「この基本設計書をもとに Arduino UNO R3 でコードを書きます。ピン競合・ライブラリ競合・状態遷移の抜け・delay() の問題点はありますか？」

**AIの回答（要約）：**
- ピン競合なし（SPI: D10/D11/D13、入力: A0/A1/D2、ブザー: D3）
- MAX7219は単色のため、色分けは点滅等で代替が必要（設計に反映済み）
- delay() を避け milllis() で周期制御する設計は妥当
- すべての状態から押し込みで復帰できるため状態遷移の抜けは少ない

**対応した内容：**
- 敵＝点滅、自機＝常時点灯で区別する設計を明記
- タイミング制御は millis() 採用を維持
---

### Q2: チャタリング・delay・ノイズの確認

> 「この設計にチャタリング対策は含まれていますか？delay() を使っている箇所で問題はありますか？アナログセンサーを使う場合、ノイズ対策は必要ですか？」

**AIの回答（要約）：**
- 押し込みSWはチャタリングが出るためデバウンス必須
- delay() は入力停止を招くため、ゲームでは millis() を推奨
- アナログ入力は中心付近の揺れがあるためデッドゾーンが有効

**対応した内容：**
- 押下エッジ＋デバウンス（50ms）を詳細設計で実装予定
- ノイズ対策としてデッドゾーン（±100〜150）を採用
