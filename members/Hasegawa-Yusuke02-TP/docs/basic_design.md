### 1-2. 状態遷移（ガジェットの「ふるまい」設計）

> ※ ガジェットが「どんな状態を持つか」を定義します。
> この設計が loop() の骨格になります。

**記入例：**
```
[電源ON / 初期化]
  ↓（初期化完了）
[待機中] ──（センサー検知）──→ [計測中] ──（閾値超え）──→ [警告中]
   ↑                                                            │
   └────────────────────（リセットボタン）────────────────────────┘
```

**↓ 自分の状態遷移を書いてください**
```
[電源ON / 初期化]
  ↓（初期化完了）
[待機状態] ──（ジョイスティック横操作）──→ [桁選択状態] ──（ジョイスティック縦操作）──→ [値変更状態]
   ↑                                           ↑                                   │
   └────────────（反映完了）← [LED出力・エフェクト状態] ←（ジョイスティック押し込み）┘
                                               └────（ジョイスティック横操作）────→ [桁選択状態]

```

**各状態の意味：**

| 状態名 | この状態のとき、何をしているか | LEDやブザーなど出力の様子 |
|:--|:--|:--|
| （例）待機中 | センサー値を100msごとに監視する | 緑LED：500msで点滅 |
| 電源ON / 初期化 | 変数やピンの初期化、ディスプレイ初期表示 | 4桁7セグメントに初期値表示、LED消灯 |
| 待機状態 | ジョイスティック操作待ち、現在値を表示 | 4桁7セグメントに値表示、LEDは前回値のまま |
| 桁選択状態 | ジョイスティック横操作で桁を移動 | 選択中の桁を点滅表示、LEDは前回値のまま |
| 値変更状態 | ジョイスティック縦操作で数字を変更 | 変更中の値を7セグメントに反映、LEDは前回値のまま |
| LED出力・エフェクト状態 | ジョイスティック押し込みで確定、LEDに反映 | 4桁の値に応じてRGB LEDの色・点滅パターンを変更 |

> [!TIP]
> すべての状態から「抜け出せるルート」があるか確認してください。
> 「このボタンを押せば必ずリセットに戻る」が保証されていますか？

### 2-3. タイミング設計（delay vs millis）

> ※ `delay(N)` はN ms間、**すべての処理**が止まります。
> 複数の処理を並行させたい場合は `millis()` を使います。

| 処理 | 必要な周期 | delay / millis | 判断の理由 |
|:--|:--|:--|:--|
| （例）LED点滅 | 500ms | **millis** | ボタン監視と並行するため |
| （例）センサー読み取り | 100ms | **millis** | 常時監視が必要なため |
| ジョイスティック読出 | 20～50ms | **millis** | 操作レスポンス向上・常時監視のため |
| 7セグ表示更新 | 50～100ms | **millis** | 入力値を即時反映・視認性向上のため |
| RGB LED出力・エフェクト | 50～500ms | **millis** | 点滅や効果を滑らかに制御するため |

> [!CAUTION]
> `delay()` を使う場合、その間はボタン入力もセンサー読み取りも**すべて止まります**。

## 7. AIレビュー記録

> グループレビューの前に必ず実施してください。

### Q1: 設計の妥当性確認

> 「この基本設計書をもとに Arduino UNO R3 でコードを書きます。ピン競合・ライブラリ競合・状態遷移の抜け・delay() の問題点はありますか？」

**AIの回答（要約）：**
ピン競合はほぼ問題なし
Arduino UNO R3で注意が必要な予約系ピンを避けた割り当てになっており、安全です。

ライブラリ競合も起きにくい
TM1637系と標準的なPWM制御の組み合わせなので、一般的な構成では衝突しにくいです。

状態遷移の抜けは見当たらない
桁選択・値変更・確定後の戻りまで流れが成立しています。
